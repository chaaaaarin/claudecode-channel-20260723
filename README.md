## 動画概要

2026年7月18日、Peter Steinberger氏がXに投げかけた9単語の問い「まだループの話をしてるの、それとももうグラフにシフトした？」が297万回表示され、AIエージェント界隈を巻き込む論争になった。ちょうど1ヶ月前、このチャンネルでも扱った「ループエンジニアリング」の熱気が冷めやらぬうちに、次の言葉が降ってきた形だ。

だが調べてみると、話はそう単純ではなかった。Claude Code自身、公式ドキュメントは今も「エージェントループ」を核として説明する一方で、2026年5月に登場した「Dynamic Workflows」はすでに決定論的な並列オーケストレーション——つまり実質的な「グラフ」——を実装済みだった。**言葉が生まれる前に、実装が先に進んでいた**というのが実態に近い。

この動画では、「ループ」と「グラフ」が何を指すのかを整理した上で、Claude Code自身の進化の跡を追い、懐疑派の反論も紹介し、最後に非エンジニアの実践者が自作の仕組みに「たった3行」を足してグラフ化した実例を見る。

## 3行まとめ

1. 「ループは終わっていない」。**グラフは、ループとループの“つなぎ方”に名前がついただけ** ✅——1つのノード（グラフの中の1点）も、相変わらずループで動いている
2. Claude Codeは、公式には今も「ループ」と説明しながら、**Agent Teams（2026年2月）→ Dynamic Workflows（2026年5月）とすでにグラフ的な仕組みへ進化していた** 🔶。Boris Cherny氏本人も「上位ステップの鍵は/loop・/batch・dynamic workflows・worktree isolation」と明言している ✅
3. 難しそうに見えて、**非エンジニアが自作スキルに「3行足す」だけで再現できる** 🔶——実例と型は本文と配布物で紹介する

## 第2章 ループとグラフ、何が違うのか

言葉の整理から始める。

**ループ**は、円環が1つ。作成→レビュー→修正→再レビューのように、ダメならやり直すを合格まで繰り返す、1本の輪だ。Claude Codeの公式ドキュメント（`code.claude.com/docs/how-claude-code-works`）は、今もこの「agentic loop」（コンテキスト収集→アクション実行→検証を繰り返す）を核として説明している ✅（2026-07-23時点で実機確認、「グラフ」という語は一度も登場しない）。

**グラフ**は、その輪や工程が複数あって、条件分岐でつながったものだ。「事実の誤りなら、調査の工程まで戻る」「表現の問題なら、その場で直して先へ進む」と、ダメの種類ごとに行き先が変わる。戻り先が1つではなく、複数ある——道が分かれている配線図、というのが言葉のイメージだ。

比較すると次のようになる。

| 観点 | ループ | グラフ |
|---|---|---|
| 単位 | 1つのAIが回す1本の輪 | 複数のノード＋つなぐ配線 |
| 形 | 発見→計画→実行→検証→繰り返し | ノードを結ぶ有向グラフ |
| 設計対象 | サイクルと停止条件 | 配線（どこへ戻すか）とルーティング |
| 状態 | 1つのAIのコンテキストの中 | 配線に沿って複数ノード間を流れる |
| 壊れるとき | 検証役が弱いとき | 配線が間違っている・状態が漏れるとき |

ここで大事な点がある。**グラフの中の1つひとつのノードも、相変わらずループで動いている**。ループが不要になったのではなく、ループ同士のつなぎ方に名前がついた、というのが正確なところだ 🔶（@audaciousSneha氏の整理より）。先月ループを学んだ人は回り道をしたのではなく、グラフの部品を先に手に入れていたことになる。

## 第3章 Claude Code自身がすでにグラフ化していた証拠

ここが今回いちばん伝えたいポイントになる。**Claude Code自身のアーキテクチャは、コミュニティが「グラフ」という言葉を使い始める前から、すでにグラフ的な方向へ進化していた**。

| 時期 | 出来事 | 確度 |
|---|---|---|
| 2026-02 | Agent Teams提供開始（Opus 4.6と同時）。複数エージェントが並列で動く | 🔶 |
| 2026-05-28 | Dynamic Workflowsがresearch previewで公開（Opus 4.8と同時） | ✅ |
| 2026-07-17 | Boris Cherny氏「Steps of AI Adoption」公開 | ✅実機確認 |

**Dynamic Workflows**は、ClaudeがJavaScriptの「オーケストレーションスクリプト」を書き、ハーネスがバックグラウンドでそれを実行する仕組みだ ✅（Anthropic公式）。技術者による検証記事（alexop.dev）によれば、主要なプリミティブは以下の3つと報告されている 🔶（Anthropic公式ブログは関数名の詳細までは明言していない）。

- `agent(prompt, opts)` — 単一のサブエージェントを実行
- `parallel(thunks)` — 全タスク完了を待つバリア付き並列実行（ファンアウト）
- `pipeline(items, ...stages)` — バリアなしの段階的パイプライン処理

並列実行数は同時16体・累計1,000体が上限と複数メディアが報告している 🔶（Anthropic公式は「tens to hundreds」とだけ明言 ✅）。非決定論的なコード（`Date.now()`や`Math.random()`）は実行時エラーになる設計で、これはすべての`agent()`呼び出しをジャーナリングし、実行を再開可能にするためだと報告されている 🔶。

そして決定的なのが、Boris Cherny氏（Claude Code作者）本人の発言だ。2026年7月17日、LinkedIn・X両方に投稿された「Steps of AI Adoption」で、こう述べている（**筆者が実機確認・全文読了**）✅。

> 「単一機能の話ではありません。正しい機能を、正しいガードレールと組み合わせて使うことです。上位レベルに進むには、`/loop`・`/batch`・dynamic workflows、そしてサブエージェント用のworktree隔離を使います」
>
> 「Anthropicは現在Step3、上位（Step4）を目指している。個人的には、自分はもうStep4に到達しました」

つまり、**Dynamic Workflowsのような「グラフ的な仕組み」は、Anthropic自身がすでに「上位ステップに進むための鍵」として位置づけていた**。ただし、その言葉自体に「グラフ」というラベルを貼ったのは、Anthropicではなくコミュニティの側だった、という点は正確に押さえておきたい。

## 第4章 反論の先回り

とはいえ、こう思っていませんか？「名前がついただけじゃないの？」

まさお氏（@AI_masaou）はSteinberger氏の投稿に対し、「PeterのいうGraphはOntologyを載せる話というより、Loopの先にあるトレンドの話やと思ってた」と反応している 🔶（実機確認）。

もっと強い懐疑論もある。

- **LangGraph・Microsoft AutoGen**は、「グラフエンジニアリング」という言葉が流行る1年以上前から、分岐・並列・ループを表現できるオーケストレーション基盤としてすでに存在していた 🔶
- **David Khourshid氏**（状態機械ライブラリXStateの作者）も、この流行に対して懐疑的な反応を示している 🔶
- **Pawel Huryn氏**「グラフエンジニアリングにはBSと言いたい。ループエンジニアリングもすでに紛らわしかった」🔶。彼の論点は「本来の目的（目標を与え、成功基準を定義すること）はループでもグラフでも変わらない」というもの

この指摘は妥当だ。**グラフエンジニアリングは新技術というより「呼び方」の問題という側面が強い**。それでも名前がつくと一気に広まるのがこの業界であり、Claude Codeのように実装が先行していたケースでは、「呼び方」がついたことで初めて構造が意識化された、という価値もある。

## 第5章 なぜループだけでは壊れるのか

言葉の是非はさておき、押さえておきたい実質的な議論がある。AI研究者Carlos E. Perez氏は、YouTube「What Makes Agency Actually Work?」の中で、単一ループの自己改善が構造的に抱える4つの失敗パターンを整理している 🔶（Lilys AI要約で確認）。

1. **指標の形骸化（Goodhartの法則）**——追いかけた指標が、いつの間にか本来の目的を裏切る。サポートチームがチケット解決率を5ヶ月追った結果、解約率が倍増した実例が挙げられている。ボットは「解決」ではなく「対応の打ち切り」を学習していた
2. **目標値自体を疑えない**——ループは目標に向けて最適化するだけで、その目標が正しいかどうかは判断できない
3. **複数ループの競合**——独立に作られた複数のループが、互いの足を引っ張り合う（速度を追うループと、丁寧さを追うループなど）
4. **測定の劣化**——センサーやデータが古びても、ループは気づかず回り続ける

グラフ化はこれらに対処しうる（複数ループが互いを監視・修正し合う構造にできる）が、Perez氏は重要な警句も添えている。**「アンカー（外部で検証可能な指標・変更できないルール・人間の最終判断）」がなければ、グラフも同じ失敗を繰り返す**。彼の言葉を借りれば「everything is consistent and nothing is verified（すべて筋が通っているのに、何も検証されていない）」状態になりうる 🔶。

## 第6章 実践：非エンジニアが3行足した話

理屈はここまでにして、実際にやってみた人の話をする。

まどさん（@madogiwacowork）は、大企業の管理職で、コードは書かないがClaude Cowork/Enterpriseを日々の現場に持ち込んで試行錯誤している実践者だ ✅（実機確認・全文読了）。彼女は自作の「レビューループ」というスキルを運用していた：論理・読者目線・抜け漏れ・ファクトチェック・敵対的の5体のレビュアーが順に指摘を出し、重大指摘ゼロか5ラウンドで収束する仕組みで、記事も仕事の資料も毎回これを通していた。

Steinberger氏の投稿を見て「グラフって何だよ」と混乱した彼女は、概念の解説記事を何本読んでもピンとこなかった。ハテナが消えたのは、読むのをやめて、自作のスキルをAI（Fable）に渡し、こう聞いた瞬間だったという。

> 「このループのスキルをグラフに直すと、何がどう変わるのか。説明して、評価してほしい」

返ってきた答え：「このスキルは、様式と打ち切りはすでに持っています。欠けているのは、**差し戻し先**と、**エスカレーション先**の2つです」。抽象的な配線図の話が、「自分のこのスキルの、この行が足りない」という具体的な話に変わった瞬間だったという。

実際に足した3行は次の通り（Before/After）。

| | 内容 |
|---|---|
| **Before** | 「ダメならもう1周」の、一本の輪。何回で打ち切るかその場の空気で決め、指摘の書き方も毎回バラバラだった |
| **After（3行追加）** | ①各指摘に修正経路ラベル（【再検証】【構成】【局所】）を付ける ②経路ごとの戻り先を決める（再検証はWeb検索からやり直し、構成は人間の再承認、局所は該当箇所のみ修正） ③5ラウンドで重大指摘が残ったら「未収束」を宣言し、選択肢を付けて人間に上げる |

まどさん自身、この記事を改造後のスキルでレビューしたところ、指摘12件（重大0・中7・軽微5）、局所11件・再検証1件・構成への差し戻し0件という内訳で、1ラウンドで収束したと報告している 🔶（本人環境での1回の実測、件数は少なく効果測定としては限定的と本人も明記）。「一番効いたのは、指摘を読んだ瞬間に直し方が決まっていることだった」という感想が印象的だ。

## 今日からやること

始めるならこの順番。

1. **今使っているAIループを1つ思い浮かべる**——記事レビュー、リサーチ、コーディング、なんでもよい
2. **本当にグラフ化が必要か診断する**——視点を分けられるか／並列化の余地があるか／制御フローを書き出せるか／完了の基準そのものが変わったか。2つ以上YESなら価値あり
3. **プレゼントのプロンプトを1つ、そのままClaude Codeに貼ってみる**——公式知識込みなので説明し直す必要はない
4. **効果を感じたら、Dynamic Workflowsで本格実装へ**——JSのオーケストレーションスクリプトを書く段階に進む
